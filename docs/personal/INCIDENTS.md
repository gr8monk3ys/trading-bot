# Incident Log

> A dated log of every weird thing that happens with the bot. Crashes, unexpected trades, broker issues, configuration mistakes, security scares — all of it. Even the small stuff.
>
> Why log everything? Because patterns emerge over months. The fourth time you see the same kind of issue, you'll wish you'd written down the first three.

## How to Use This File

When something happens, add a new entry at the **top** of the incidents section below. Use the template format. Be specific. Future-you will thank present-you.

If the incident is severe (real money lost, security breach, regulatory issue), also:
1. Open a GitHub issue with label `incident`
2. Update `RUNBOOK.md` if the response procedure needs improvement
3. Add to `DECISIONS.md` if the incident drove a major architectural change

## Severity Levels

- **P0 (Critical):** Real money lost, account compromised, illegal activity, bot doing something dangerous. Stop everything. Investigate immediately.
- **P1 (High):** Bot crashed during market hours, missed expected trades, unauthorized access attempt detected. Fix same day.
- **P2 (Medium):** Strategy underperforming expectations, deployment glitches, non-critical errors. Fix this week.
- **P3 (Low):** Cosmetic issues, minor log noise, documentation gaps. Fix when convenient.

## Entry Template

YYYY-MM-DD — [Short title]
Severity: P0 / P1 / P2 / P3
What happened: Plain description of the symptom. What did you observe?
When: Date, time (with timezone), how long it lasted.
Impact: Money lost, trades missed, downtime, security exposure. Be specific. "Bot was down" is not impact — "bot was down for 2 hours during market open, missed 3 expected trades, no positions affected" is impact.
Root cause: Why did this happen? If unknown, say "unknown" — don't fabricate.
Resolution: What did you do to fix it? Be specific enough that you could repeat the fix.
Prevention: What change (code, runbook, monitoring, process) would prevent recurrence?
Follow-up actions: GitHub issues created, runbook updates, dependency upgrades, etc. Link them.
Status: Resolved / In progress / Won't fix / Recurring (link prior incidents)

## 2026-05-03 — Backtest mode silently drops all strategy orders

**Severity:** P1 (High) — blocks Phase 1 validation entirely

**What happened:** First-ever backtest on the forked codebase appeared to complete successfully but produced zero trades and 0.00% return regardless of strategy, symbols, or date range. The bot loaded data, ran through all 251 trading days of 2024 in seconds, and reported "Backtest complete" with no errors at the summary level.

Closer inspection of the run logs revealed every order generated by the strategy was silently rejected:

Trade signal: AAPL - short
ERROR - No OrderGateway configured. Entry order blocked; gateway-only routing is mandatory.
SHORT 5 shares of AAPL @ $173.72

The strategy was correctly generating signals but every order was being blocked at the strategy's `submit_entry_order` guard in `strategies/base_strategy.py`.

**When:** 2026-05-03, during initial Phase 0/Phase 1 transition. First backtest attempt after Alpaca paper credentials were configured.

**Impact:** Phase 1 validation was completely blocked. Without working backtests, strategy logic couldn't be validated and the project couldn't progress past tooling setup. No real money exposure (we never reached live trading), but several hours of debugging effort.

**Root cause:** A regression introduced by upstream commit `feat: enforce live validation evidence gates (#22)`. That commit added mandatory OrderGateway routing for live trading (a good change), but didn't update the backtest path to either:

1. Pass `enforce_gateway=False` to the `StrategyManager` in backtest mode, or
2. Inject an OrderGateway into strategies when `BacktestEngine` instantiates them

Two specific gaps:

- `main.py` (lines 522 and 1302) creates `StrategyManager(broker=broker)` without `enforce_gateway=False`. Default of `True` propagates to the `OrderGateway`, which blocks all orders in backtest because no real broker is wired up.

- `engine/backtest_engine.py` instantiates strategies inside `run_backtest` with `strategy_class(broker=backtest_broker, parameters=params)` — no `order_gateway` argument. So strategies have `self.order_gateway = None`, which fails the guard in `BaseStrategy.submit_entry_order` regardless of any other configuration.

The error message itself ("No OrderGateway configured. Entry order blocked") was clear once we found it, but it appeared in INFO/ERROR log lines that didn't propagate to the user-facing summary — backtests appeared to "succeed" with zero trades.

**Resolution:**

Two-part fix in PR `https://github.com/AchyuthPilly/trading-bot/pull/3` (or whatever number) on this fork:

1. `main.py`: pass `enforce_gateway=False` to `StrategyManager` in `run_backtest()` (line 522) and `optimize_parameters()` (line 1302).
2. `engine/backtest_engine.py`: create a scoped `OrderGateway(broker=backtest_broker, enforce_gateway=False)` immediately before strategy instantiation, and pass it via `order_gateway=backtest_gateway` kwarg. Includes a `try/except TypeError` fallback for strategy classes that don't accept the kwarg.

**Verification:**

uv run python main.py backtest --strategy MomentumStrategyBacktest 
--symbols AAPL,MSFT,GOOGL,NVDA,META,AMZN 
--start-date 2024-01-01 --end-date 2024-12-31 --capital 10000

Before fix: 0 trades, 0% return.
After fix: 6 orders submitted, +21.26% return, Sharpe 1.62, max drawdown 2.53%.

**Prevention:** A few things would have caught this earlier:

- A smoke test in CI that runs a tiny backtest and asserts `Number of Trades > 0` for at least one canonical strategy/date range. This exact bug class (silent order rejection) would fail that test.
- The strategy guard's `logger.error("No OrderGateway configured...")` should arguably propagate up to a thrown exception instead of just logging and returning `None`. Silent dropping of orders is dangerous.
- The summary printer should differentiate "0 trades because no signals" from "0 trades because all orders were rejected" — the user-visible output should make rejection rate visible.

**Follow-up actions:**

- ✅ Fix shipped in PR `#3`.
- 📋 Phase 3 task: properly wire `StrategyManager.order_gateway` through to `BacktestEngine` rather than creating a separate gateway. Right now we have two gateways floating around in backtest mode (one in StrategyManager, unused; one we create in BacktestEngine, used). Cleaner to thread the same instance through.
- 📋 Phase 3 task: investigate the `WARNING - Could not check positions: object list can't be used in 'await' expression` from `utils/order_gateway.py`. Likely caused by `BacktestBroker.get_positions()` returning a list directly while `OrderGateway` expects an awaitable.
- 📋 Add a smoke-test backtest to CI to prevent regression.

**Status:** Resolved. Root cause understood, fix verified, follow-ups tracked.

**Lessons learned:**

- Silent order rejection is a dangerous failure mode. Even in paper trading, a backtest that says "completed: 0 trades" with no warning at the summary level is misleading. Defensive code should prefer loud failures over quiet zeros.
- AI-assisted upstream commits (this regression came from the original repo's recent enforcement work) can introduce subtle but high-impact bugs. Forking captures snapshots in time — review and test the fork before relying on it.
- The strategy/gateway architecture is sound but the wiring is incomplete in backtest mode. This will be a theme to watch in Phase 3 hardening.
